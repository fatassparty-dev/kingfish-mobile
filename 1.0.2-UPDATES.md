# KingFish iPhone — 1.0.2 updates

**Build:** `com.kingfishbets.app` 1.0.2 (build 15) — built & shipped to TestFlight 2026-06-25.
**Why a build at all:** the fix is client code compiled into the bundle, and the app
has **no OTA / expo-updates** — it cannot be fixed from the server.

---

## The bug (what users saw)

On 1.0.1, after launch/sign-in the app would freeze: **premium account shown as
"Free"**, **name/location "Not set"**, **player props stuck "Loading…" forever**, and
the **Sign Out button did nothing** (a dead button Brian had flagged repeatedly
during 1.0.1). It did **not** reliably self-heal. **It reproduced in the iOS
Simulator**, not just on a device.

## Root cause (verified in library source AND reproduced/fixed in the Simulator)

A **re-entrant Supabase auth-lock deadlock.**

`@supabase/auth-js` fires `onAuthStateChange` **while holding its internal auth
lock**. Our `lib/auth.tsx` handler did `await loadProfile()` →
`kingfishFetch('/api/account')` → `getAccessToken()` → **`supabase.auth.getSession()`**,
and `getSession()` needs that **same lock** → it waits on a lock the current call
stack already holds → **deadlock** (until `lockAcquireTimeout`).

That single deadlock froze everything at once:
- profile read never completes → **premium shows "Free", name/location "Not set"**;
- every data fetch's `getAccessToken()`→`getSession()` hangs → **props spin forever**;
- `signOut()` also needs the lock → **Sign Out button does nothing**.

Because it's a **logic** deadlock, not a network one, it reproduces **in the
Simulator** — which is the fact that finally disproved the wrong theory below.
Most likely introduced by the *"read entitlement from `/api/account`"* change — that's
what made profile-loading call `getSession()` from inside the auth callback.

### ⚠️ Wrong theory chased first (recorded so we don't repeat it)
A lot of time went into an **HTTP/3 (QUIC) stall** theory: that `setSession()`'s
client-side `GET /auth/v1/user` to `supabase.co` (Cloudflare, advertises `alt-svc:
h3`) stalled on device. The "fix" was routing mobile Supabase through
`kingfishbets.com/supabase-proxy` (HTTP/2) + a 15s `kingfishFetch` timeout. **That did
NOT fix the freeze** — it reproduced in the Simulator, which has no QUIC. Lesson:
**when a bug reproduces in the Simulator, it's client logic, not the network.** The
proxy + timeout were kept only as **hardening** (see below).

## The fix (commit `d6c434b`)

`lib/auth.tsx`:
1. **Defer the `onAuthStateChange` body** with `setTimeout(…, 0)` so the auth lock is
   released **before** `loadProfile`/`getSession` runs. This is the supabase-documented
   way to avoid this exact deadlock (don't call other `supabase.auth` methods inside
   the callback).
2. **Sign Out no longer `await`s a lock-bound `signOut()`.** A hang is not a throw, so
   the previous `try/catch` never recovered and the button looked dead. It now fires
   `signOut({ scope: 'local' })` without awaiting and clears state + navigates
   immediately, so the button always responds.

**Verified in the iOS Simulator and on-device via TestFlight:** premium shows, props
load, Sign Out works.

### Also in this build (commit `f29f44e`)
- **Hardening (not the fix):** mobile Supabase routed via `kingfishbets.com/supabase-proxy`
  (`.env` + `eas.json`) and a 15s timeout in `kingfishFetch` (`lib/api.ts`). Kept as
  insurance against a separate possible device-only HTTP/3 stall; harmless, proven to
  return 200 with real Supabase JSON.
- **Sign-up Location is now a dropdown** matching the web's `LOCATION_OPTIONS`
  (Other/outside US + states, Puerto Rico after Pennsylvania). Files:
  `app/(auth)/sign-up.tsx`, `lib/locations.ts`.
- **Plain-language weak-password error** instead of the raw Supabase message that
  dumped the full character sets. File: `app/(auth)/sign-up.tsx`.
- Version bump to **1.0.2 / 15** (`app.json` + local `ios/` Info.plist & pbxproj).

## Build & ship recipe (the working flow — EAS not used)

Version is in `app.json`, `ios/KingFishBets/Info.plist` (literal strings), and
`ios/KingFishBets.xcodeproj/project.pbxproj` (`MARKETING_VERSION` /
`CURRENT_PROJECT_VERSION`). `.env` and `ios/` are gitignored. Team ID **3275YRB2Q7**,
cert "Apple Distribution: KingFish Bets, LLC". Do **not** use Xcode GUI Run/Archive
(project has no GUI dev team set). From Terminal:

```
# 1) export options
cat > ~/Desktop/ExportOptions.plist  # method=app-store, teamID=3275YRB2Q7, signingStyle=automatic, uploadSymbols=true
# 2) unsigned release archive (avoids "requires a development team")
cd .../kingfish-mobile/ios && xcodebuild -workspace KingFishBets.xcworkspace -scheme KingFishBets \
  -configuration Release -sdk iphoneos -archivePath ~/Desktop/KingFishBets-1.0.2.xcarchive \
  archive CODE_SIGNING_ALLOWED=NO
# 3) export signed .ipa
xcodebuild -exportArchive -archivePath ~/Desktop/KingFishBets-1.0.2.xcarchive \
  -exportOptionsPlist ~/Desktop/ExportOptions.plist -exportPath ~/Desktop/KingFishBets-1.0.2-export
# 4) drag the .ipa into the Transporter app → Deliver (do NOT double-click the .ipa)
```

## Loose ends found during this investigation (separate from the fix)
- **No Sentry email alerts** despite live App Hang / crash issues — an alert-rule
  config gap. Set up notifications.
- **`Cannot find native module 'Visionocr'`** in Sentry — the Grade-My-Slip OCR module
  may not be linked in some build; verify it loads on device. Track independently.
- **Sentry symbolication:** App Hang stacks show only native frames (no JS). Upload
  Hermes source maps so future hangs name the JS function.
- **iPad/Mac (kingfish-studio) likely shares this deadlock pattern** — being checked
  (read-only) since it was forked from the same auth code. See that app's notes.
