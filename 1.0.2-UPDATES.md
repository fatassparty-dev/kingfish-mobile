# KingFish iPhone — 1.0.2 updates

**Build target:** `com.kingfishbets.app` 1.0.2 (next build number after 14)
**Date opened:** 2026-06-25
**Why a build at all:** the fix is client code compiled into the bundle, and the
app has **no OTA / expo-updates** — so it cannot be fixed from the server. See
"Could this be server-side only?" below for the full reasoning (it can't).

---

## The bug (what users saw)

On 1.0.1, **after signing in** the app would sit with **no data** — player props
stuck "loading," account shown as **Free**, **no name**, **no location** — and it
**never recovered** (30s+), including "Restore Purchases." Sometimes a full app
restart cleared it; **after a fresh sign-in, a restart did not.** Web was fine.

## Root cause (confirmed from library source, not a guess)

The app authenticates through a custom server endpoint (`POST /api/auth/sign-in`),
which mints a session and returns `{ access_token, refresh_token }`. The client
then calls `supabase.auth.setSession(...)` to install that session locally.

In `@supabase/auth-js` **2.105.1**, for a *fresh* (non-expired) token, `setSession()`
**acquires the auth lock and then makes a client-side network call** —
`GET /auth/v1/user` — and only fires `SIGNED_IN` **after** it returns
(`node_modules/@supabase/auth-js/dist/module/GoTrueClient.js`, `_setSession`).

The mobile Supabase client pointed **directly at `gkfyuobwnzwannbrzklc.supabase.co`**
(Cloudflare), which advertises **`alt-svc: h3=":443"`** (HTTP/3 / QUIC). On device,
that `/auth/v1/user` request **QUIC-stalls** — the exact failure that previously
bricked the KingFish Studio Mac Catalyst build (TLS handshake completes, then idle
timeout, no fallback to HTTP/2).

Because `setSession()` is **holding the auth lock** while that call hangs:

- `SIGNED_IN` never fires → `loadProfile()` never runs → **no `/api/account` call**
  (verified absent in Vercel logs) → no profile → "Free / no name / no location."
- Every other auth call (`getAccessToken()`, the profile retry loop, Restore
  Purchases) **blocks on the same held lock** → total auth deadlock, no self-heal.

**Why a restart fixed the update case but not the sign-in case:** on relaunch the
app reads a previously-saved session via `getSession()` (no `/user` call) and works.
But `setSession()` saves the session **after** the hung `/user` call, so a fresh
sign-in **never persists** — relaunch returns to sign-in and stalls again.

### What we ruled out first (so the fix targets the real cause)
- **Token race / bad token:** no — Supabase auth logs are all `200`, fast (token
  ~20–120ms). Sign-in itself succeeds.
- **Slow/cold server (`/api/account` latency):** no — when reached it's a fast 200.
- **Large-payload `JSON.parse` blocking the JS thread:** no — payloads are tiny
  (`/api/account` 366 B, `/api/mlb-schedule` 25 KB). Hypothesis dropped.
- The server-side auth logs looked fast because they are **Vercel→Supabase**. The
  stalling call is **device→Supabase**, a path those logs never covered — which is
  why this took a while to localize.

## The fix

Point the mobile Supabase client at the existing same-origin proxy on
`kingfishbets.com`, which is **HTTP/2-only and serves no `alt-svc` h3**, so every
client→Supabase call (including `setSession()`'s `/auth/v1/user`) speaks HTTP/2 and
cannot QUIC-stall. `kingfish-bets/next.config.ts` already rewrites
`/supabase-proxy/* → https://…supabase.co/*` (added for Studio's identical stall).

**Safe for mobile:** the app uses Supabase for **auth only** — `0` `supabase.from()`
REST calls, no Realtime/websockets — so a rewrite covers 100% of its Supabase usage.

### Files changed (all client; 1.0.2)
- **`.env`** — `EXPO_PUBLIC_SUPABASE_URL` → `https://kingfishbets.com/supabase-proxy`.
  *(Authoritative for the build: the iPhone IPA is an unsigned `xcodebuild` archive,
  and Expo inlines `EXPO_PUBLIC_*` from `.env` at bundle time. EAS is not used.)*
- **`eas.json`** (`build.production.env`) — same URL change, kept in sync so the two
  sources don't drift if anyone ever builds via EAS.
- **`lib/api.ts`** — `kingfishFetch` now attaches a **15s default timeout** (when the
  caller hasn't supplied its own `signal`). Defense-in-depth: a future hung request
  now **errors → React Query retries**, instead of spinning "loading" forever. This
  does **not** fix the auth deadlock (that's the proxy's job) — it hardens the data
  paths against any other stall.

## Considered and deferred (not in this build)
- **`AppState` start/stop-auto-refresh wiring** for the Supabase client (Supabase RN
  best practice). Low risk, but not the cause here; revisit separately.
- **Tightening the profile retry ladder** (`PROFILE_RETRY_DELAYS_MS`, currently
  ~12s worst case) to shorten any residual "Free" flash. The proxy makes the first
  read succeed, so this is now a rare fallback — left as-is to keep this build tight.
- **A timeout/guard around `setSession()` in `sign-in.tsx`** so a hung auth call can
  never lock the UI even if some other edge stalls. Reasonable hardening; deferred to
  keep the change surface minimal and focused on the confirmed root cause.

## Verification before submitting
1. Build 1.0.2 and **test in a real TestFlight build, not `⌘R`** (the QUIC behavior
   only reproduces on a real device/network — same lesson as Studio).
2. **Sign in fresh** (delete app first) on cellular and on Wi-Fi → data, name,
   location, and premium status should appear within ~1–2s.
3. Confirm Vercel logs show `GET /api/account` (200) firing right after
   `POST /api/auth/sign-in`, and the Supabase `/auth/v1/user` traffic now arrives via
   the `/supabase-proxy` rewrite (HTTP/2, `server: Vercel`).
4. Restore Purchases works post-sign-in.

## Loose ends found during this investigation (separate from the fix)
- **No Sentry email alerts** despite live "App Hang"/crash issues — an **alert-rule
  config gap**. Set up notifications so the next regression isn't found by a user.
- **`Cannot find native module 'Visionocr'`** in Sentry — the Grade-My-Slip OCR
  native module isn't linked in the iPhone build. Separate bug; track independently.
- **Sentry symbolication** for the mobile app: App Hang stacks show only native
  frames (no JS), making them hard to act on. Upload Hermes source maps for 1.0.2 so
  future hangs name the JS function.
